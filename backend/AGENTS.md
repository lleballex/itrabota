# AGENTS.md

## Назначение

Этот файл задаёт короткие рабочие правила для Codex в backend-репозитории.

Подробные описания архитектуры, доменной модели, API и принятых решений должны жить в отдельных документах в `docs/` и быть source of truth для деталей.

## Базовый принцип

Перед тем как фиксировать новые архитектурные правила, соглашения или паттерны в документации:

1. сначала проанализировать существующий код;
2. затем показать вывод пользователю;
3. только после согласования обновлять документацию.

Если в коде обнаружено несколько конфликтующих или расходящихся паттернов, нельзя молча выбирать один из них как правильный. Нужно явно зафиксировать расхождение и вынести вопрос пользователю.

Если правило нельзя надёжно вывести из кода или согласованных документов, его нельзя додумывать.

## Структура модулей

Backend организован как NestJS modular monolith.

Для модуля обязательным считается только файл `*.module.ts`.

Остальные части модуля добавляются только если это требуется конкретной предметной областью:

- `*.controller.ts`
- `*.service.ts`
- `dto/`
- `entities/`
- `guards/`
- `decorators/`
- `strategies/`
- `interfaces/`
- `types/`

Если для дополнительной структуры есть потребность, новые папки и типы файлов нужно хранить консистентно с уже существующими модулями.

## Repo-specific conventions

- Сначала опираться на фактический код, а не на предположения.
- Основной паттерн: `controller -> service -> TypeORM repository/entity`.
- Сущность можно инжектить только в сервис её собственного модуля.
- Из сервиса нельзя напрямую работать с сущностью другого модуля через `@InjectRepository(...)`.
- Доступ к данным другого модуля должен идти через его сервис.
- Для простых операций ориентироваться на существующий стиль методов:
- `findAll`
- `findOneById`
- `create`
- `update`
- `delete` / `remove`
- DTO хранить в `dto/`, сущности в `entities/`, локальные типы в `types/`, auth-related инфраструктуру в `decorators/`, `guards/`, `strategies/`, `interfaces/`.
- Импорты файлов текущего модуля делать через относительные пути, например `./`.
- Импорты файлов из других модулей и общих областей делать через alias `@/`.
- Не распространять случайные несогласованности и опечатки из текущего кода. Это технический долг, а не норма.
- Если нужен новый модульный элемент, сначала проверить, как такие элементы уже расположены в соседних модулях.
- Для новых ошибок предпочитать стандартные HTTP exceptions NestJS.
- Сообщения в HTTP exceptions писать на английском языке.
- Если в коде встречаются обычные `Error`, не считать это целевым паттерном. При затрагивании соответствующего места не расширять этот подход без отдельного согласования.
- Swagger plugin считается основным источником генерации схем.
- Явные swagger-декораторы добавлять только там, где plugin не справляется автоматически.
- Если пришлось добавить swagger-декоратор, нужно понимать, что именно plugin не вывел сам, и при необходимости зафиксировать это в документации по API.

## Работа с документацией

`AGENTS.md` должен содержать только долговечные и часто применимые инструкции.

Подробности должны выноситься в отдельные документы:

- `docs/architecture.md` — source of truth по архитектурным паттернам и границам модулей
- `docs/domain-model.md` — source of truth по сущностям и связям
- `docs/api-contracts.md` — source of truth по маршрутам, DTO и ограничениям API

Если текущий код противоречит документации, нужно:

1. считать код фактическим источником текущего состояния;
2. показать расхождение пользователю;
3. не переписывать документацию без согласования.

После изменений нужно отдельно проверить, требуется ли обновление документации.

Если документацию действительно нужно менять, сначала нужно показать это пользователю и согласовать обновление.

## Команды проверки проекта

Основные команды:

- `yarn build` — сборка проекта
- `yarn lint` — проверка ESLint
- `yarn prettier --check .` — проверка форматирования
- `yarn prettier --write .` — исправление форматирования при необходимости

## Что обязательно проверять после изменений

После изменений нужно проверить:

- проект собирается: `yarn build`
- нет ошибок типов TypeScript
- нет ошибок ESLint
- Prettier не сообщает о проблемах форматирования
- при необходимости запустить Prettier и привести изменённые файлы к принятому формату
- если затронуты DTO, контроллеры, guards или auth-flow, нужно отдельно проверить, что не сломаны валидация, роли и форма входных данных
- если затронуты сущности или связи TypeORM, нужно проверить, что остаются корректными сервисы, query builder-запросы, загрузка связей и ожидаемая форма ответа
- если затронут Swagger-контракт, нужно проверить, не требуется ли явный swagger-декоратор там, где plugin не выводит схему автоматически
- если затронута составная операция в нескольких сервисах или сущностях, нужно отдельно оценить риск частичного сохранения, потому что в проекте есть места без транзакций
- после изменений нужно проверить, требуется ли обновление документации, и не менять её без согласования с пользователем

## Перед завершением задачи

В итоговом сообщении нужно отдельно указать:

- что было изменено
- какие проверки были выполнены
- какие проверки не удалось выполнить
- требуется ли обновление документации
- какие неоднозначности или архитектурные расхождения были обнаружены
