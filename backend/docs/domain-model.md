# Доменная модель backend

## Назначение документа

Этот документ фиксирует доменную модель backend-проекта в том виде, в котором её можно надёжно вывести из кода.

Документ описывает:
- сущности;
- их назначение;
- ключевые поля;
- связи;
- ограничения и инварианты, которые явно видны в коде;
- места, где бизнес-смысл нельзя уверенно восстановить без дополнительного уточнения.

Если какое-то правило не подтверждается кодом, оно здесь не формулируется как факт.

## Технические сущности и общие замечания

### BaseEntity

Тип: техническая сущность.

Назначение:
- общая базовая абстракция для всех entity-классов.

Ключевые поля:
- `id: string`
- `createdAt: Date`
- `updatedAt: Date`

Ограничения и инварианты:
- `id` генерируется как UUID;
- все сущности наследуют одинаковые технические поля аудита.

Неочевидно:
- дополнительные общие поля сейчас отсутствуют, но если базовая структура модели потребует нового общего поля, его нужно будет добавлять в `BaseEntity`;
- `createdAt` и `updatedAt` могут использоваться не только для хранения, но и в бизнес-логике.

## Доменные сущности

### User

Назначение:
- базовая учётная запись пользователя системы;
- хранит данные для аутентификации и роль;
- связывает пользователя с профилем кандидата или рекрутера.

Ключевые поля:
- `email: string`
- `password?: string`
- `role: UserRole`
- `passwordChangedAt: Date | null`

Связи:
- `candidate?: Candidate | null` — one-to-one
- `recruiter?: Recruiter | null` — one-to-one

Ограничения и инварианты:
- `email` уникален;
- `password` по умолчанию не выбирается из БД;
- `role` обязателен;
- пользователь может не иметь заполненного профиля кандидата или рекрутера;
- по доменной логике роль определяет допустимый связанный профиль:
  - для `candidate` должен использоваться профиль кандидата;
  - для `recruiter` должен использоваться профиль рекрутера.
- пользователь может зарегистрироваться и долго не заполнять профиль;
- для значительной части функционала наличие профиля всё же требуется;
- по бизнес-правилу у пользователя не должно быть двух профилей одновременно, хотя чисто технически это сейчас не запрещено.

Неочевидно и требует уточнения:
- текущий набор ролей считается фактическим, но теоретически в будущем могут появиться дополнительные роли, например `admin` или владелец компании без профиля.

### Candidate

Назначение:
- профиль пользователя с ролью кандидата;
- хранит личные данные, навыки, город, опыт работы, аватар и отклики.

Ключевые поля:
- `firstName: string`
- `lastName: string`
- `patronymic: string | null`
- `bornAt: Date`
- `phoneNumber: string | null`
- `tgUsername: string | null`
- `description: string | null`

Связи:
- `city?: City | null` — one-to-one по факту использования сущности города как ссылки
- `skills?: Skill[]` — many-to-many
- `workExperience?: WorkExperienceItem[]` — one-to-many
- `avatar?: Attachment | null` — one-to-one
- `user?: User` — one-to-one
- `applications?: Application[]` — one-to-many

Ограничения и инварианты:
- `firstName`, `lastName`, `bornAt` обязательны;
- `user` обязателен;
- удаление `User` каскадно удаляет `Candidate`;
- город и аватар могут отсутствовать;
- список навыков и опыт работы могут отсутствовать;
- отклики кандидата привязаны к профилю кандидата, а не напрямую к `User`.
- `phoneNumber` и `tgUsername` необязательны;
- по бизнес-правилу одному пользователю должен соответствовать только один профиль кандидата.

Неочевидно и требует уточнения:
- дополнительные бизнес-ограничения на дату рождения сейчас не реализованы;
- `description` — это текст кандидата о себе для рекрутеров.

### Recruiter

Назначение:
- профиль пользователя с ролью рекрутера;
- хранит персональные данные рекрутера и связывает его с компанией и вакансиями.

Ключевые поля:
- `firstName: string`
- `lastName: string`
- `patronymic: string | null`

Связи:
- `user?: User` — one-to-one
- `company?: Company` — one-to-one
- `vacancies?: Vacancy[]` — one-to-many

Ограничения и инварианты:
- `firstName` и `lastName` обязательны;
- `user` обязателен;
- `company` обязательна на уровне связи в entity;
- удаление `User` каскадно удаляет `Recruiter`;
- удаление `Recruiter` каскадно влияет на связанные вакансии и компанию через их связи.
- по бизнес-правилу у рекрутера должна быть ровно одна компания;
- несколько рекрутеров на одну компанию в текущем домене не допускаются.

Неочевидно и требует уточнения:
- связь `Recruiter -> Company` является осознанным упрощением для текущей версии продукта;
- в случае развития продукта модель теоретически может быть расширена, но сейчас это не планируется.

### Company

Назначение:
- компания, с которой связан рекрутер;
- хранит базовую карточку компании и логотип.

Ключевые поля:
- `name: string`
- `url: string | null`

Связи:
- `logo?: Attachment | null` — one-to-one
- `recruiter?: Recruiter` — one-to-one
- `industry?: Industry` — many-to-one

Ограничения и инварианты:
- `name` обязателен;
- `industry` обязателен;
- `recruiter` обязателен;
- `logo` может отсутствовать;
- удаление `Industry` каскадно удаляет `Company`;
- удаление `Recruiter` каскадно удаляет `Company`.
- компания в текущей модели концептуально принадлежит одному рекрутеру;
- `Company` считается самостоятельной доменной сущностью, а не просто вложенной структурой профиля.

Неочевидно и требует уточнения:
- валидация `url` как настоящего URL сейчас отсутствует и рассматривается как потенциальное улучшение.

### Vacancy

Назначение:
- вакансия, опубликованная рекрутером;
- хранит описание позиции, условия, справочные классификаторы и воронку этапов.

Ключевые поля:
- `title: string`
- `description: string | null`
- `requirements: string | null`
- `niceToHave: string | null`
- `responsibilities: string | null`
- `conditions: string | null`
- `status: VacancyStatus`
- `salaryFrom: number | null`
- `salaryTo: number | null`
- `employmentType: VacancyEmploymentType`
- `format: VacancyFormat`
- `schedule: VacancySchedule`
- `workExperience: VacancyWorkExperience`

Связи:
- `skills?: Skill[]` — many-to-many
- `specialization?: Specialization` — many-to-one
- `city?: City | null` — many-to-one
- `recruiter?: Recruiter` — many-to-one
- `funnelSteps?: FunnelStep[]` — one-to-many
- `applications?: Application[]` — one-to-many

Ограничения и инварианты:
- `title` обязателен;
- `status` обязателен, по умолчанию `active`;
- `employmentType`, `format`, `schedule`, `workExperience` обязательны;
- `specialization` обязательна;
- `recruiter` обязателен;
- `city` может отсутствовать;
- зарплатные поля могут отсутствовать;
- навыки являются необязательной many-to-many связью;
- удаление `Recruiter` каскадно удаляет `Vacancy`;
- удаление `Specialization` каскадно удаляет `Vacancy`;
- удаление `City` не удаляет вакансию, а зануляет ссылку.
- вакансия может существовать без воронки;
- допустимые статусы вакансии сейчас ограничены теми, что уже описаны в коде;
- вакансия точно может архивироваться;
- физическое удаление вакансии как бизнес-сценарий пока не определено.

Неочевидно и требует уточнения:
- ограничения `salaryFrom <= salaryTo` сейчас нет;
- обязательность `city` для офисного формата сейчас не enforced.

### FunnelStep

Назначение:
- шаг воронки обработки откликов по вакансии.

Ключевые поля:
- `index: number`
- `name: string`
- `approveMessage: string | null`
- `rejectMessage: string | null`
- `shouldCreateCall: boolean`

Связи:
- `vacancy?: Vacancy` — many-to-one
- `applications?: Application[]` — one-to-many

Ограничения и инварианты:
- `name` обязателен;
- `vacancy` обязательна;
- `index` обязателен, по умолчанию `0`;
- `shouldCreateCall` по умолчанию `false`;
- удаление `Vacancy` каскадно удаляет `FunnelStep`.
- по смыслу `index` должен задавать порядок шагов внутри вакансии;
- `approveMessage` и `rejectMessage` используются как шаблоны сообщений при принятии и отклонении отклика;
- `shouldCreateCall` означает, что при выборе соискателем даты и времени нужно автоматически создавать видеовстречу.

Неочевидно и требует уточнения:
- по смыслу `index` должен быть уникален внутри вакансии, но это сейчас не видно как enforced-ограничение;
- шаги не обязаны идти без пропусков и не обязаны начинаться с нуля, важен сам порядок.

### Application

Назначение:
- отклик кандидата на вакансию;
- хранит статус отклика, связь с кандидатом, вакансией, текущим этапом и сообщениями.

Ключевые поля:
- `status: ApplicationStatus`

Связи:
- `candidate?: Candidate` — many-to-one
- `vacancy?: Vacancy` — many-to-one
- `funnelStep?: FunnelStep | null` — many-to-one
- `messages?: ApplicationMessage[]` — one-to-many

Ограничения и инварианты:
- `candidate` обязателен;
- `vacancy` обязательна;
- `status` обязателен, по умолчанию `pending`;
- `funnelStep` может отсутствовать;
- по сервисной логике нельзя создать второй отклик того же кандидата на ту же вакансию;
- удаление `Candidate` или `Vacancy` каскадно удаляет `Application`;
- удаление `FunnelStep` зануляет ссылку на текущий этап.
- после создания отклик может перейти в `approved` или `rejected`;
- после завершения отклик остаётся видимым в истории, но дальнейшие действия над ним не предполагаются;
- текущая модель допускает отсутствие `funnelStep`.

Неочевидно и требует уточнения:
- текущий набор статусов считается достаточным, но в будущем может измениться;
- должно ли приложение всегда иметь `funnelStep` после определённого момента, пока не определено;
- коммуникация сейчас существует только в рамках отклика, отдельный домен коммуникации пока не планируется.

### ApplicationMessage

Назначение:
- сообщение или системное событие внутри отклика.

Ключевые поля:
- `senderRole: UserRole`
- `type: ApplicationMessageType`
- `content: string | null`

Связи:
- `application?: Application` — many-to-one

Ограничения и инварианты:
- `senderRole` обязателен;
- `type` обязателен;
- `application` обязательна;
- `content` может отсутствовать;
- удаление `Application` каскадно удаляет `ApplicationMessage`.
- `ApplicationMessage` сейчас представляет собой смесь пользовательских сообщений и системных событий;
- на текущем этапе `senderRole` считается достаточным для определения автора.

Неочевидно и требует уточнения:
- набор `type` пока считается достаточным, но в будущем может расшириться.

### Attachment

Назначение:
- техническая сущность для хранения бинарных вложений.

Тип:
- техническая, но используется в доменных сущностях.

Ключевые поля:
- `name: string`
- `mimeType: string`
- `size: number`
- `content?: Buffer`

Связи:
- прямых обратных связей в сущности нет;
- используется как ссылка из `Candidate.avatar` и `Company.logo`.

Ограничения и инварианты:
- `name`, `mimeType`, `size` обязательны;
- `content` хранится в БД и по умолчанию не выбирается;
- `size` обязателен;
- attachment может использоваться как вложение доменной сущности.
- модель задумана как общая для любых файлов, а не только для аватаров и логотипов.

Неочевидно и требует уточнения:
- ограничения по MIME type, размеру, типу файла, а также отдельный статус или тип вложения пока не реализованы и рассматриваются как возможное будущее расширение.

### City

Назначение:
- справочник городов.

Ключевые поля:
- `name: string`

Связи:
- используется из `Candidate.city`
- используется из `Vacancy.city`

Ограничения и инварианты:
- `name` обязателен;
- город является справочником;
- дубликаты названий по бизнес-смыслу не допускаются.

Неочевидно и требует уточнения:
- регион, страна, slug и внешние идентификаторы сейчас не нужны.

### Industry

Назначение:
- справочник отраслей компаний.

Ключевые поля:
- `name: string`

Связи:
- используется из `Company.industry`

Ограничения и инварианты:
- `name` обязателен;
- отрасль является справочником;
- дубликаты названий по бизнес-смыслу не допускаются.

Неочевидно и требует уточнения:
- древовидная структура отраслей сейчас не нужна.

### Skill

Назначение:
- справочник навыков.

Ключевые поля:
- `name: string`

Связи:
- используется в `Candidate.skills`
- используется в `Vacancy.skills`

Ограничения и инварианты:
- `name` обязателен;
- навык является элементом справочника;
- дубликаты названий и разные варианты написания одного и того же навыка по бизнес-смыслу не допускаются.

Неочевидно и требует уточнения:
- категории навыков сейчас не нужны.

### Specialization

Назначение:
- справочник специализаций вакансий.

Ключевые поля:
- `name: string`

Связи:
- используется в `Vacancy.specialization`

Ограничения и инварианты:
- `name` обязателен;
- специализация является справочником направления вакансии;
- по смыслу специализация отличается:
  - от отрасли компании;
  - от навыков кандидата;
- дубликаты названий по бизнес-смыслу не допускаются.

Неочевидно и требует уточнения:
- отрасль описывает сферу компании, навыки описывают компетенции кандидата, специализация описывает направление вакансии;
- иерархическая модель специализаций сейчас не нужна.

### WorkExperienceItem

Назначение:
- отдельная запись опыта работы кандидата.

Ключевые поля:
- `position: string`
- `companyName: string`
- `startedAt: string`
- `endedAt: string | null`
- `description: string | null`

Связи:
- `candidate?: Candidate` — many-to-one

Ограничения и инварианты:
- `position`, `companyName`, `startedAt` обязательны;
- `candidate` обязателен;
- `endedAt` может отсутствовать;
- удаление `Candidate` каскадно удаляет элементы опыта работы.
- `endedAt = null` означает, что работа продолжается;
- опыт работы по смыслу должен иметь порядок отображения, но в текущей модели явное поле сортировки не видно.

Неочевидно и требует уточнения:
- ограничение `endedAt >= startedAt` логически ожидаемо, но сейчас явно не реализовано;
- необходимость отдельного хранения типа занятости или других атрибутов опыта сейчас не подтверждена;
- порядок опыта работы по текущему коду неочевиден.

## Доменные перечисления и словари, которые не являются сущностями

Ниже перечислены важные доменные значения, которые участвуют в модели, но не оформлены как отдельные entity-классы.

### UserRole

Назначение:
- роль пользователя в системе.

Значения, видимые в коде:
- `recruiter`
- `candidate`

Неочевидно:
- набор ролей пока считается достаточным, но потенциально может измениться.

### VacancyStatus

Значения, видимые в коде:
- `active`
- `archived`

Неочевидно:
- других статусов сейчас не предусмотрено, но список может расшириться.

### VacancyEmploymentType

Значения, видимые в коде:
- `full_time`
- `part_time`
- `internship`

### VacancyFormat

Значения, видимые в коде:
- `office`
- `remote`
- `hybrid`

### VacancySchedule

Значения, видимые в коде:
- `flexible`
- `five_two`
- `six_one`
- `two_two`
- `three_three`
- `four_two`
- `other`

### VacancyWorkExperience

Значения, видимые в коде:
- `none`
- `up_to_year`
- `one_to_three_years`
- `three_to_five_years`
- `from_five_years`

### ApplicationStatus

Значения, видимые в коде:
- `pending`
- `rejected`
- `approved`

Неочевидно:
- список пока считается достаточным, но в будущем может измениться;
- при необходимости статус можно будет вынести и экспортировать как отдельный публичный словарь.

### ApplicationMessageType

Значения, видимые в коде:
- `user`
- `candidate_responded`

Неочевидно:
- набор типов пока считается достаточным, но в будущем может измениться.

## Что логично было бы добавить или зафиксировать, но сейчас этого нет

Ниже перечислены вещи, которые по смыслу выглядят полезными или ожидаемыми, но сейчас не подтверждены как реализованные ограничения или правила:

- дополнительные общие поля в `BaseEntity`, если когда-нибудь станет ясно, что они нужны всей модели;
- дополнительные бизнес-ограничения на дату рождения кандидата;
- валидация `Company.url` как настоящего URL;
- проверка `salaryFrom <= salaryTo` у вакансии;
- обязательность `city` для офисного формата вакансии;
- техническое или доменное ограничение уникальности `FunnelStep.index` внутри одной вакансии;
- ограничение `endedAt >= startedAt` у `WorkExperienceItem`;
- явная сортировка опыта работы кандидата;
- ограничения по MIME type, размеру и типу файла для `Attachment`;
- отдельный статус или тип вложения для `Attachment`;
- возможность администратора управлять справочниками;
- более строгое разделение бизнес-инвариантов и UI-валидации.

## Общие вопросы по доменной модели

По коду нельзя надёжно вывести без твоего уточнения:
- какие ограничения должны считаться бизнес-инвариантами, а какие пока только UI-валидацией;
- справочники сейчас не меняются пользователями и администраторами, но по хорошему администратор в будущем должен иметь возможность управлять всеми справочниками.
