# Архитектура backend

## Назначение документа

Этот документ фиксирует core pattern разработки backend-проекта.

Цель документа:

- сохранить фактическую архитектурную модель проекта;
- зафиксировать согласованный эталонный паттерн для новых изменений;
- не допускать появления новых несогласованных структур без явного согласования.

Если код расходится с этим документом, нужно сначала определить, является ли это:

- фактическим текущим состоянием;
- техническим долгом;
- новым паттерном, который ещё не согласован.

## Назначение backend-архитектуры

Backend построен как NestJS modular monolith.

Архитектура проекта организована вокруг предметных модулей. Каждый модуль обычно завязан на основной доменной сущности и инкапсулирует:

- HTTP-слой, если модуль опубликован в API;
- application/service-логику;
- доступ к собственной сущности через TypeORM;
- локальные DTO, типы и вспомогательные структуры, если они нужны именно этому модулю.

Главная цель такой архитектуры:

- группировать код по предметным областям;
- держать логику рядом с её основной сущностью;
- не размазывать работу с одной сущностью по разным модулям;
- управлять межмодульными зависимостями через сервисы, а не через прямой доступ к чужим сущностям.

## 1. Фактическое текущее состояние

### 1.1. Принципы модульности

Фактически проект состоит из NestJS-модулей в `src/modules`.

По коду видно:

- у каждого предметного блока есть свой `*.module.ts`;
- модуль может содержать контроллер, один или несколько сервисов, DTO, сущности и дополнительные подпапки;
- не каждый модуль имеет контроллер;
- не каждый модуль имеет одну сущность;
- часть модулей является API-модулями, часть инфраструктурными или внутренними.

Примеры:

- `vacancies`, `auth`, `me` имеют контроллеры;
- `users`, `companies`, `applications` сейчас не имеют собственных контроллеров;
- `me` и `users` содержат несколько сервисов;
- `auth` содержит `decorators`, `guards`, `interfaces`, `strategies`;
- `users` содержит `entities` и `types`.

### 1.2. Что считается модулем

Фактически модулем считается директория в `src/modules/<module-name>` с файлом `<module-name>.module.ts`.

На текущий момент модуль обычно назван по основной сущности или группе сущностей в множественном числе:

- `users`
- `vacancies`
- `applications`
- `attachments`
- `skills`
- `industries`
- `specializations`
- `cities`
- `companies`

Отдельно существует функциональный модуль `auth` и модуль `me`, который агрегирует логику текущего пользователя.

### 1.3. Фактическая структура модуля

По текущему коду можно уверенно вывести:

- обязательный файл модуля: `<module-name>.module.ts`
- остальные файлы и папки опциональны и появляются по необходимости

Встречающиеся элементы структуры:

- `<module-name>.controller.ts`
- один или несколько `*.service.ts`
- `dto/`
- `entities/`
- `decorators/`
- `guards/`
- `strategies/`
- `interfaces/`
- `types/`

Важно:

- состав модуля сейчас не единообразен;
- отсутствие папки или файла не считается нарушением;
- наличие новой папки или нового типа файла само по себе не запрещено, но такие расширения должны согласовываться с пользователем.

### 1.4. Фактический слой доступа к данным

Основной паттерн в коде:

- модуль работает со своей сущностью через `Repository<Entity>`;
- `Repository` инжектится в сервис собственного модуля через `@InjectRepository(...)`;
- сервисы других модулей подключаются через DI и используются как граница межмодульного доступа.

Также в коде фактически встречается следующее:

- простые чтения делаются через `findOne`, `findOneBy`, `find`;
- более сложные выборки делаются через `QueryBuilder`;
- в части сервисов есть приватный `findOne(...)` и публичные обёртки;
- в части сервисов есть `create(...)`, `update(...)`, иногда `delete/remove(...)`.

При этом текущее состояние не полностью единообразно:

- не в каждом сервисе есть `createQB`;
- не в каждом сервисе есть приватный `findOne`;
- не все сервисы используют единый шаблон методов;
- не везде ошибки оформлены через HTTP exceptions;
- в коде есть технический долг в именовании и структуре отдельных файлов.

### 1.5. Фактические правила нейминга

По текущему коду можно вывести следующие фактические правила:

- модуль: `<module-name>.module.ts`
- контроллер: `<module-name>.controller.ts`
- основной сервис модуля: `<module-name>.service.ts`
- дополнительные сервисы: `<specific-part>.service.ts`
- DTO: `CreateXxxDto`, `UpdateXxxDto`, `GetXxxDto`
- entity: `Xxx`
- папки DTO и entity: `dto/`, `entities/`
- локальные типы: `types/`

Но единообразие в репозитории сейчас нарушено техническим долгом. Примеры:

- `skills.entity.ts` при классе `Skill`
- `update-vacancy-dto.ts` вместо более консистентного `update-vacancy.dto.ts`
- `work-experence-item.entity.ts` с опечаткой

Эти случаи нельзя считать нормой.

### 1.6. Фактические правила именования методов

В контроллерах и сервисах для собственной сущности обычно используются короткие общие названия без имени сущности:

- `findAll`
- `findOne`
- `findOneById`
- `create`
- `update`

Когда метод относится к другому контексту или нужен дополнительный смысл, в имени появляется уточнение:

- `findAllForRecruiter`
- `findAllForCandidate`
- `findOneForAuth`
- `findOneForCurCandidate`

Это наблюдается и в сервисах, и в контроллерах.

### 1.7. Фактическое разделение логики сервисов

По текущему коду видно несколько вариантов:

1. Простой сервис вокруг одной сущности.
   Признаки:

- один `Repository`
- базовые методы чтения/создания/обновления
- минимум orchestration-логики

2. Сервис-оркестратор внутри модуля.
   Признаки:

- использует несколько сервисов;
- координирует создание или обновление нескольких связанных сущностей;
- часто содержит приватные методы для валидации и upsert-логики.

3. Модуль с несколькими сервисами вместо одного большого.
   Признаки:

- отдельный сервис на основную сущность;
- отдельные сервисы на связанные части домена;
- агрегирующий модульный слой над ними.

Единственный жёстко соблюдаемый паттерн по текущему коду вывести нельзя, но общий вектор прослеживается: сущность и её базовые операции стараются держать в собственном сервисе.

### 1.8. Фактическая работа с DTO

По коду подтверждается:

- входные DTO размещаются внутри модуля в `dto/`;
- DTO используются в контроллерах;
- DTO валидируются через `class-validator`;
- вложенные DTO описываются через `@Type(...)` и `@ValidateNested(...)`;
- для create/update применяются отдельные DTO-классы;
- DTO могут содержать идентификаторы связанных сущностей, а не сами сущности.

Swagger plugin используется как основной способ построения схем.

Из пользовательского уточнения следует:

- swagger-декораторы обычно не нужны;
- если `ApiProperty` или другие swagger-декораторы всё же добавлены, значит plugin не смог корректно вывести схему автоматически;
- такие случаи нужно рассматривать как специальные, а не как базовый паттерн.

### 1.9. Что фактически лежит в shared/core/common

Сейчас в `src/common` находятся только общие utility types.

Сейчас в `src/database` находится общая базовая сущность `BaseEntity`.

Из этого можно надёжно вывести:

- в shared/common выносятся только действительно общие и повторно используемые вещи;
- domain-специфичные DTO, сервисы, сущности и типы туда не выносятся;
- модульная логика в основном остаётся внутри своего модуля.

## 2. Эталонный паттерн для новых изменений

Этот раздел фиксирует не просто наблюдение, а уже согласованный ориентир для новых изменений.

### 2.1. Принципы модульности

Эталонный подход:

- каждый модуль должен быть привязан к основной сущности;
- название модуля должно быть в формате `<сущность во множественном числе>`;
- модуль должен оставаться основной точкой входа в работу со своей сущностью;
- доступ к данным другой сущности должен идти через сервис её модуля, а не через прямой доступ к её репозиторию.

Если предполагается модуль не вокруг сущности, а вокруг функциональности, это нужно отдельно согласовывать с пользователем.

### 2.2. Что считается модулем

Эталонно модуль:

- лежит в `src/modules/<plural-entity-name>/`;
- содержит `<plural-entity-name>.module.ts`;
- может содержать любые дополнительные файлы и папки, если они действительно нужны;
- расширение структуры модуля новыми типами файлов или папок должно согласовываться с пользователем.

### 2.3. Обязательная и допустимая структура модуля

Обязательно:

- `<module-name>.module.ts`

Допустимо по необходимости:

- `<module-name>.controller.ts`
- `<module-name>.service.ts`
- дополнительные `*.service.ts`
- `dto/`
- `entities/`
- `decorators/`
- `guards/`
- `strategies/`
- `interfaces/`
- `types/`
- другие папки и файлы, если они действительно нужны и согласованы

Нельзя считать обязательным наличие:

- контроллера;
- одной единственной сущности;
- одного единственного сервиса;
- фиксированного набора подпапок.

### 2.4. Правила нейминга

Эталонный паттерн:

- модуль: `<entities>.module.ts`
- папка модуля: имя основной сущности во множественном числе
- контроллер: `<entities>.controller.ts`
- основной сервис: `<entities>.service.ts`
- вспомогательный сервис: `<specific-domain-part>.service.ts`
- entity-класс: `Xxx`
- entity-файл: `xxx.entity.ts`
- DTO-класс: `CreateXxxDto`, `UpdateXxxDto`, `GetXxxDto`
- DTO-файл: `create-xxx.dto.ts`, `update-xxx.dto.ts`, `get-xxx.dto.ts`

Если текущее имя в коде не соответствует этому паттерну, это нужно считать техническим долгом, а не новым правилом.

### 2.5. Правила импортов

Эталонный паттерн:

- импорт файлов текущего модуля делать через относительные пути, например `./` и `../`;
- импорт файлов из других модулей и общих областей делать через alias `@/`.

### 2.6. Правила именования методов контроллеров

Для работы с собственной сущностью контроллера методы должны называться без имени сущности:

- `findAll`
- `findOne`
- `create`
- `update`
- `delete`

Имя сущности в названии метода используется только если:

- метод работает с другой сущностью;
- без уточнения теряется смысл;
- нужно явно обозначить конкретный контекст.

Примеры допустимых уточнений:

- `findAllForRecruiter`
- `createApplication`
- `findMyApplication`

### 2.7. Правила разделения методов сервисов

Эталонный паттерн сервиса для собственной сущности:

- всегда должен быть метод `createQB(...)`, который создаёт `QueryBuilder` и инкапсулирует общие joins, sorting и базовые настройки;
- должен быть приватный метод `findOne(where, options?)`, который использует `createQB(...)`, получает сущность и выбрасывает `404`, если запись не найдена;
- публичные методы получения сущности должны быть обёртками вокруг `findOne(...)` и называться по смыслу, например:
  - `findOneById`
  - `findOneForAuth`
  - `findOneForRecruiter`
- должны быть методы, если требуются:
  - `create(data: DeepPartial<Entity>)`
  - `update(id: string, data: DeepPartial<Entity>)`
- если требуется удаление, допустимы `delete(...)` или `remove(...)` по смыслу текущего сервиса.

Важно:

- даже внутри собственного сервиса не нужно каждый раз обращаться к репозиторию напрямую, если уже есть универсальный внутренний метод, который решает ту же задачу;
- повторно используемая логика выборки должна идти через `createQB(...)` и `findOne(...)`, а не копироваться по сервису;
- сужение типов в публичных обёртках допустимо.

### 2.8. Шаблон эталонного сервиса

```ts
@Injectable()
export class EntitiesService {
  constructor(
    @InjectRepository(Entity)
    private readonly entitiesRepo: Repository<Entity>,
  ) {}

  private createQB() {
    return this.entitiesRepo.createQueryBuilder("entity")
    // общие join'ы, сортировка и базовые настройки
  }

  private async findOne(
    where: FindOptionsWhere<Entity>,
    options?: {
      // дополнительные параметры при необходимости
    },
  ) {
    const qb = this.createQB().setFindOptions({ where })

    const entity = await qb.getOne()

    if (!entity) {
      throw new NotFoundException("Entity not found")
    }

    return entity
  }

  findOneById(id: string) {
    return this.findOne({ id })
  }

  async create(data: DeepPartial<Entity>) {
    const entity = this.entitiesRepo.create(data)
    const savedEntity = await this.entitiesRepo.save(entity)

    return this.findOneById(savedEntity.id)
  }

  async update(id: string, data: DeepPartial<Entity>) {
    const entity = this.entitiesRepo.create({ id, ...data })
    await this.entitiesRepo.save(entity)

    return this.findOneById(id)
  }
}
```

Это именно эталонный паттерн для новых сервисов. Текущий код ещё не везде приведён к нему.

### 2.9. Правила работы с DTO

Эталонный паттерн:

- DTO хранить внутри модуля в `dto/`;
- create и update разделять на разные DTO;
- query DTO выносить в отдельные `GetXxxDto`;
- DTO должны описывать входной контракт, а не внутреннюю структуру сущности;
- связанные сущности в DTO передавать через `id`, если код не требует другой формы;
- вложенные структуры описывать отдельными DTO-классами внутри файла или рядом, если это оправдано;
- swagger-декораторы добавлять только если plugin сам не строит нужную схему.

### 2.10. Что можно выносить в shared/core/common

Можно выносить:

- общие utility types;
- действительно общие базовые сущности;
- общие технические абстракции, которые не принадлежат одному доменному модулю.

Не нужно выносить:

- доменные сущности конкретного модуля;
- модульные DTO;
- сервисы, относящиеся к одной предметной области;
- типы, которые используются только в одном модуле;
- логику, которая нужна только одной сущности.

Если появляется ощущение, что что-то стоит вынести в `common`, нужно сначала проверить:

- используется ли это минимум в нескольких модулях;
- не размывает ли это границы домена;
- не станет ли shared-папка свалкой несвязанных абстракций.

## 3. Anti-patterns и признаки плохой структуры

К anti-patterns для этого проекта относятся:

- прямой `@InjectRepository(...)` для сущности другого модуля;
- обход сервиса другого модуля ради доступа к его данным;
- дублирование логики выборки вместо использования `createQB(...)` и `findOne(...)`;
- именование методов вида `findUser`, `createVacancy`, `updateCandidate` внутри сервиса или контроллера собственной сущности, когда контекст и так очевиден;
- распространение опечаток и несогласованных имён как будто это норма;
- вынос модульной логики в `common` без реальной повторной применимости;
- создание новых папок и структур внутри модуля без необходимости и без согласования;
- фиксация архитектурного правила в документации без анализа текущего кода и согласования с пользователем;
- молчаливый выбор одного паттерна при наличии нескольких конфликтующих реализаций в коде;
- использование обычного `Error` как нормы для новых сценариев вместо стандартных HTTP exceptions;
- сообщения в HTTP exceptions не на английском языке;
- повторные прямые обращения к репозиторию внутри собственного сервиса там, где уже существует универсальный сервисный метод.

## 4. Что считать техническим долгом, а не правилом

Не считать архитектурной нормой:

- случайные опечатки в именах файлов и методов;
- локальные несогласованности нейминга;
- использование обычного `Error` в части сервисов;
- отсутствие единого сервисного шаблона в старом коде;
- любые отклонения, которые не подтверждены как согласованный паттерн.

## 5. Как использовать этот документ

При добавлении нового модуля или изменении существующего:

1. сначала сравнить задачу с фактическим текущим кодом;
2. затем проверить, подпадает ли решение под эталонный паттерн из этого документа;
3. если подпадает, следовать ему;
4. если не подпадает или требует новой структуры, сначала согласовать это с пользователем;
5. если в коде найден конкурирующий паттерн, не выбирать молча, а зафиксировать расхождение и вынести вопрос пользователю.
