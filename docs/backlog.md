# Backlog

## 1. Как читать backlog

- Backlog сформирован на основе:
  - текущего состояния проекта;
  - целевого scope;
  - gap-analysis.
- Одна задача должна быть достаточно конкретной, чтобы по ней можно было работать отдельно.
- Типы задач:
  - `feature`
  - `bugfix`
  - `refactor`
  - `integration`
  - `ux`
  - `tech debt`
- Приоритеты:
  - `критичный`
  - `высокий`
  - `средний`
  - `низкий`
- Примерная сложность:
  - `S`
  - `M`
  - `L`

## 2. Критический минимум

1. Исправить контракт создания отклика между frontend и backend.
2. Добавить тип процесса `response|invitation` в `Application`.
3. Реализовать отправку приглашения рекрутером кандидату.
4. Реализовать единый раздел процессов с табами `Отклики/Приглашения`.
5. Реализовать принятие приглашения кандидатом.
6. Реализовать отклонение процесса кандидатом.
7. Реализовать отклонение кандидата рекрутером с причиной.
8. Реализовать перевод кандидата по этапам воронки.
9. Реализовать раздел кандидатов для рекрутера.
10. Добавить флаг видимости резюме кандидата и учёт этого флага в поиске/приглашениях.
11. Реализовать backend и frontend архивации/восстановления вакансий.
12. Добавить пагинацию обязательных списков.
13. Добавить `responsesCount` в список вакансий рекрутера.
14. Реализовать сущность `Meeting`.
15. Реализовать выбор времени встречи кандидатом.
16. Реализовать календари кандидата и рекрутера с отображением встреч.
17. Реализовать in-app уведомления через popover.
18. Добавить обязательные системные сообщения в историю процесса.

## 3. Важные, но не критичные

1. Довести auth-конфигурацию и env-настройки.
2. Усилить валидацию и согласованность форм кандидата.
3. Доработать update-flow вакансий по связанным полям.
4. Добавить базовую стабилизацию транзакций в составных операциях.
5. Вынести localhost-хардкоды в конфиг.
6. Довести профиль рекрутера и компании по техдолгу.
