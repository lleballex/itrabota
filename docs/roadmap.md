# Roadmap

## 1. Цель roadmap

- Определить последовательность этапов разработки до состояния, когда проект можно считать минимально готовым по согласованному `target scope`.
- Учитывать зависимости между задачами, а не просто перечислять backlog подряд.
- Выделить критический путь до минимально готовой версии.
- Отделить post-MVP задачи, которые можно отложить после дедлайна.

## 2. Ограничения и допущения

- Целевой горизонт: `1-2 недели`.
- Основная часть реализации предполагается через агента, с ревью и кураторством со стороны человека.
- Главный риск по срокам находится во frontend-блоках, где нужен новый интерфейс:
  - раздел кандидатов;
  - раздел процессов;
  - календари;
  - уведомления.
- Для попадания в срок ранние этапы должны использовать существующие UI-паттерны и не уходить в новый сложный дизайн.
- Инфраструктурный baseline и техдолг внедряются по мере разработки, а не как отдельный большой стартовый этап.

## 3. Критический путь

1. Стабилизировать текущую базу:
   - исправить контракт отклика;
   - убрать критичные хардкоды;
   - стабилизировать update-flow вакансии.
2. Построить правильную модель процесса:
   - `Application.type`;
   - единый раздел процессов;
   - accept/reject flows;
   - системная история.
3. Закрыть recruiter-side сценарий кандидатов:
   - флаг видимости резюме;
   - раздел кандидатов;
   - приглашение кандидата.
4. Реализовать рабочую воронку:
   - перевод по этапам;
   - связь этапов с историей.
5. Реализовать встречи:
   - `Meeting`;
   - выбор времени;
   - автогенерация ссылки;
   - календари.
6. Реализовать in-app уведомления:
   - сущность;
   - popover;
   - генерация по событиям.
7. Довести обязательные списки и вакансии:
   - пагинация;
   - архивирование / восстановление;
   - `responsesCount`.

Если хотя бы один из пунктов критического пути не завершён, минимально готовую версию нельзя считать полностью собранной.

## 5. Этапы реализации

### 5.1 Этап 0. Стабилизация базы

**Цель**

- Убрать очевидные поломки и минимально подготовить проект к безопасной доработке.

**Какие задачи входят**

- Исправить контракт создания отклика между frontend и backend.
- Вынести localhost-хардкоды в конфиг.
- Довести auth-конфигурацию и env-настройки.
- Доработать update-flow вакансии по связанным полям.
- По возможности точечно добавить транзакции в самые рискованные составные операции без попытки закрыть весь техдолг.

**Какой результат должен быть получен**

- Текущий код перестаёт ломать базовые сценарии.
- Появляется безопасная основа для расширения `Application`, `Vacancy`, `Candidate`.
- Снижается риск накопления новых расхождений между фронтом и бэком.

**Что блокирует следующий этап**

- Если не исправлен контракт отклика, нельзя надёжно строить историю процесса.
- Если update-flow вакансии остаётся нестабильным, рискованно развивать воронку и recruiter flow.

**Какие есть риски**

- Уход в слишком большой рефакторинг.
- Попытка закрыть весь техдолг до продуктовых задач.
- Потеря времени на инфраструктуру вместо движения по критическому пути.

**Оценка по сроку**

- `1-2 дня`

### 5.2 Этап 1. Базовая модель процесса найма

**Цель**

- Превратить текущий отклик в полноценную основу для единого процесса найма.

**Какие задачи входят**

- Добавить тип процесса `response|invitation`.
- Реализовать единый раздел процессов с табами `Отклики/Приглашения`.
- Добавить пагинацию в список процессов.
- Реализовать отклонение процесса кандидатом.
- Реализовать отклонение кандидата рекрутером с причиной.
- Добавить системные сообщения для ключевых событий процесса.
- Реализовать общий список процессов по вакансии.

**Какой результат должен быть получен**

- И кандидат, и рекрутер получают единый центр управления процессами.
- История процесса становится рабочей доменной частью продукта.
- В проекте появляется правильная база для invitation-flow и дальнейшей воронки.

**Что блокирует следующий этап**

- Пока нет общего процесса и его истории, нельзя нормально строить recruiter flow кандидатов и приглашений.
- Пока нет `type`, отклики и приглашения невозможно прозрачно поддерживать как одну сущность.

**Какие есть риски**

- Слишком раннее усложнение статусов процесса.
- Попытка сделать полноценный чат вместо системной истории.
- Перегрузка UI раздела процессов до завершения базовой логики.

**Оценка по сроку**

- `2-3 дня`

### 5.3 Этап 2. Recruiter flow: кандидаты и приглашения

**Цель**

- Закрыть обязательный recruiter-side сценарий поиска кандидатов и отправки приглашений.

**Какие задачи входят**

- Добавить флаг видимости резюме кандидата.
- Доработать форму кандидата и профиль под этот флаг.
- Реализовать раздел кандидатов для рекрутера.
- Реализовать просмотр профиля кандидата рекрутером.
- Реализовать backend-flow приглашения кандидата.
- Добавить action приглашения из recruiter UI.
- Добавить пагинацию списка кандидатов.

**Какой результат должен быть получен**

- Рекрутер может искать кандидатов с открытым резюме.
- Рекрутер может открыть профиль кандидата и отправить приглашение на вакансию.
- Кандидат может скрыть резюме из поиска, но продолжать откликаться сам.
- Приглашения становятся полноценной частью продукта.

**Что блокирует следующий этап**

- Без recruitерского сценария кандидатов продукт не соответствует обязательному scope.
- Без приглашений встречи и календарь будут покрывать только часть обязательного процесса.

**Какие есть риски**

- Этот этап потребует нового UI и с высокой вероятностью потребует ручной доработки человеком.
- Есть риск расползания задачи в большой каталог кандидатов, который сейчас не нужен.

**Оценка по сроку**

- `2-3 дня`

### 5.4 Этап 3. Воронка, встречи и календарь

**Цель**

- Замкнуть обязательный процесс найма от приглашения/отклика до этапа со встречей и отображения результата в календарях.

**Какие задачи входят**

- Реализовать перевод кандидата по этапам воронки.
- Ввести сущность `Meeting`.
- Реализовать выбор времени встречи кандидатом.
- Реализовать автогенерацию ссылки встречи.
- Встроить события встреч в историю процесса.
- Реализовать календарь кандидата.
- Реализовать календарь рекрутера.

**Какой результат должен быть получен**

- Рекрутер управляет этапами процесса.
- Если этап требует встречи, кандидат выбирает слот.
- Создаётся отдельная сущность встречи.
- Встреча видна в истории процесса и в календарях обеих ролей.

**Что блокирует следующий этап**

- Без встреч и календарей проект не соответствует обязательному target scope.
- Пока этот этап не завершён, продукт нельзя считать готовым.

**Какие есть риски**

- Это самый тяжёлый этап по объёму новой логики.
- Есть риск уйти в сложный календарный UX, который сейчас не нужен.
- Есть риск преждевременно начать делать расписание рекрутера, переносы и расширенное управление календарём.

**Оценка по сроку**

- `3-4 дня`

### 5.5 Этап 4. Уведомления и доведение MVP

**Цель**

- Довести продукт до состояния, когда обязательный сценарий не только работает, но и пригоден для повседневного использования.

**Какие задачи входят**

- Реализовать сущность и API уведомлений.
- Реализовать popover уведомлений.
- Привязать доменные события к генерации уведомлений.
- Добавить `responsesCount` в список вакансий рекрутера.
- Реализовать архивацию вакансий.
- Реализовать восстановление вакансий.
- Завершить пагинацию обязательных списков, если часть не закрыта раньше.
- Довести профиль рекрутера и компании по техдолгу.
- Довести мелкие UX-пробелы в обязательных сценариях.

**Какой результат должен быть получен**

- Пользователь получает уведомления о ключевых событиях.
- У рекрутера список вакансий выглядит завершённым.
- Архив вакансий реально работает.
- Все обязательные списки масштабируются через пагинацию.
- Обязательный scope закрыт до состояния минимально готовой версии.

**Что блокирует следующий этап**

- Этот этап завершает минимально готовую версию, а не подготавливает следующий этап.

**Какие есть риски**

- Опасность распылиться на полировку.
- Попытка превратить popover уведомлений в отдельный крупный модуль.
- Перенос в этот этап лишних post-MVP задач.

**Оценка по сроку**

- `2 дня`

### 5.6 Этап 5. Post-MVP

**Цель**

- Закрыть всё, что важно, но не входит в критический путь до готовой версии.

**Какие задачи входят**

- Удаление вакансий.
- Пользовательские сообщения в процессе.
- Email-уведомления.
- Публичный каталог компаний.
- Matching / степень соответствия.
- Сложные фильтры по кандидатам.
- Сложные фильтры по процессам.
- Расширенные фильтры по вакансиям.
- Настройка расписания рекрутера.
- Перенос и отмена встреч.
- Админ-панель.
- Импорт резюме через ИИ.
- Полный переход на миграции, если он не будет закрыт раньше по ходу разработки.

**Какой результат должен быть получен**

- Продукт становится богаче и удобнее без изменения core-MVP статуса.

**Что блокирует следующий этап**

- Не блокирует готовность MVP.

**Какие есть риски**

- Если начать эти задачи до завершения этапов 0-4, дедлайн почти наверняка будет сорван.
- Некоторые задачи имеют высокую неопределённость: matching, AI import, admin analytics.

## 7. Риски и контрольные точки

### Основные риски

- Новый UI может стать бутылочным горлышком:
  - раздел кандидатов;
  - раздел процессов;
  - календари;
  - popover уведомлений.
- Слишком ранний уход в полировку:
  - сложные фильтры;
  - чат;
  - email-уведомления;
  - matching;
  - улучшения календаря.
- Слишком большой объём backend-изменений без фиксации модели:
  - `Application`;
  - `Meeting`;
  - системные сообщения;
  - переходы по воронке.
- Попытка закрыть весь техдолг до фич:
  - это замедлит движение по критическому пути.

### Контрольные точки

- После этапа 0:
  - текущий функционал не содержит явных критичных расхождений в базовых сценариях.
- После этапа 1:
  - есть единый процесс найма и история ключевых событий.
- После этапа 2:
  - recruiter-side сценарий кандидатов и приглашений работает.
- После этапа 3:
  - воронка, встречи и календари работают end-to-end.
- После этапа 4:
  - обязательный scope закрыт до минимально готовой версии.
