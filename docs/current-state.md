# Current State

## 1. Scope и методика аудита

- Аудит выполнен по фактическому коду `frontend` и `backend`, без попытки додумывать отсутствующий функционал.
- Оценка строится по пользовательским функциональным областям проекта, а не по внутренним модулям в отрыве от сценариев.
- Для каждой области фиксируются:
  - статус;
  - что реально подтверждается кодом;
  - что реализовано только частично;
  - признаки недоделанности, нестабильности или технического долга.
- Если степень готовности нельзя надёжно вывести из кода, это явно отмечается как неопределённость.
- Если между frontend и backend есть расхождение, оно фиксируется отдельно.

## 2. Карта проекта

### 2.1 Общая структура

- `frontend/` — клиентское приложение на `Next.js App Router`.
- `backend/` — API и доменная логика на `NestJS + TypeORM`.
- Проект фактически разделён на два самостоятельных приложения с общими предметными областями:
  - аутентификация;
  - профили пользователя;
  - вакансии;
  - отклики;
  - справочники;
  - вложения.

### 2.2 Frontend

- `src/app` — маршруты и page-level composition.
- `src/components/base` — доменные UI-блоки.
- `src/components/ui` — низкоуровневые примитивы.
- `src/components/special` — app-level поведение, включая `AuthProvider` и `Toasts`.
- `src/api` — hooks и HTTP-интеграции с backend.
- `src/types` — типы доменных сущностей.

### 2.3 Backend

- `src/modules/auth` — регистрация, вход, выход, JWT cookie auth.
- `src/modules/me` — работа с профилем текущего пользователя.
- `src/modules/vacancies` — вакансии и шаги воронки.
- `src/modules/applications` — отклики и сообщения в откликах.
- `src/modules/attachments` — хранение и отдача файлов.
- `src/modules/cities`, `industries`, `specializations`, `skills` — справочники.
- `src/modules/users`, `companies` — внутренняя доменная логика без самостоятельного пользовательского UI.

## 3. Функциональные области

### 3.1 Аутентификация и доступ

- Статус: `частично реализовано`
- Оценка:
  - На backend есть `register`, `login`, `logout`, получение текущего пользователя через `/api/me`, JWT в cookie, guards и role-based auth.
  - На frontend есть страницы логина и регистрации, redirect-логика через `AuthProvider`, блокировка доступа по ролям и проверка наличия профиля.
- Что подтверждается кодом:
  - пользователь может зарегистрироваться;
  - пользователь может войти;
  - пользователь может выйти;
  - клиент умеет запрашивать текущего пользователя и ограничивать доступ к страницам.
- Почему не выглядит завершённым:
  - `logout` не защищён авторизацией;
  - cookie options выглядят незавершёнными для production;
  - CORS жёстко привязан к `http://localhost:3000`;
  - часть конфигурации завязана на TODO.
- Признаки техдолга / нестабильности:
  - нет завершённой env-конфигурации для origin/cookie;
  - нет отдельного подтверждения production-ready security settings.

### 3.2 Профиль кандидата

- Статус: `частично реализовано`
- Оценка:
  - Есть создание и обновление профиля кандидата.
  - Фронт покрывает основные поля: ФИО, дата рождения, город, email, контакты, описание, аватар, опыт работы.
  - Backend хранит больше доменной информации, чем реально доступно пользователю на фронте.
- Что подтверждается кодом:
  - создание профиля кандидата;
  - обновление профиля кандидата;
  - загрузка / удаление аватара;
  - создание, обновление и удаление записей опыта работы;
  - привязка города;
  - изменение email пользователя из экрана профиля.
- Почему не выглядит завершённым:
  - backend поддерживает `skillIds`, но frontend не даёт редактировать навыки;
  - даты и телефон на frontend валидируются как обычные строки, без явного предметного контроля;
  - нет отдельного подтверждения бизнес-правил для формата даты и контактов.
- Признаки техдолга / нестабильности:
  - составные операции выполняются без транзакции;
  - часть валидации на frontend помечена TODO;
  - есть опечатка в имени файла компонента `CanidateProfileWorkExperience.tsx`.

### 3.3 Профиль рекрутера

- Статус: `частично реализовано`
- Оценка:
  - Реализованы создание и обновление рекрутера и компании.
  - Пользователь может редактировать ФИО, email, компанию, сайт, индустрию и логотип.
- Что подтверждается кодом:
  - создание профиля рекрутера;
  - обновление профиля рекрутера;
  - создание / замена / удаление логотипа компании;
  - привязка компании к индустрии.
- Почему не выглядит завершённым:
  - блок смены пароля является UI-заглушкой без сценария и API;
  - backend помечает проверку существования индустрии как TODO;
  - логика обработки логотипа на frontend содержит временный обходной код.
- Признаки техдолга / нестабильности:
  - составные операции без транзакции;
  - экран пароля недоделан;
  - есть явные TODO в сервисе и во frontend submit-flow.

### 3.4 Справочники

- Статус: `частично реализовано`
- Оценка:
  - На backend есть города, индустрии, специализации и навыки.
  - На frontend реально используются города, индустрии и специализации.
  - Навыки как справочник на backend есть, но полноценного пользовательского сценария на фронте не видно.
- Что подтверждается кодом:
  - `GET /cities`;
  - `GET /industries`;
  - `GET /specializations`;
  - `GET /skills`.
- Почему не выглядит завершённым:
  - frontend для городов и специализаций подменяет ошибку API мок-данными;
  - навыки не подключены в пользовательские формы, хотя предметно поддерживаются на backend.
- Признаки техдолга / нестабильности:
  - моковые fallback-данные скрывают реальные проблемы API;
  - отсутствие фронтового сценария для навыков создаёт разрыв между моделью данных и UI.

### 3.5 Вакансии

- Статус: `частично реализовано`
- Оценка:
  - У рекрутера есть создание, редактирование, список и просмотр вакансий.
  - У кандидата есть список вакансий и просмотр карточки вакансии.
  - Настройка воронки найма встроена в форму вакансии.
- Что подтверждается кодом:
  - `GET /api/vacancies/recruiter`;
  - `GET /api/vacancies/candidate`;
  - `POST /api/vacancies`;
  - `GET /api/vacancies/:id`;
  - `PATCH /api/vacancies/:id`;
  - frontend-экраны для списка, просмотра, создания и редактирования вакансии.
- Почему не выглядит завершённым:
  - в UI есть кнопки архивации и возврата из архива, но нет действия и нет backend endpoint;
  - поле `responsesCount` ожидается frontend-карточкой вакансии, но backend его не формирует;
  - update-flow вакансии по связям стабилизирован (явный mapping `specializationId`, `cityId`, `skillIds` и update `funnelSteps`), но блок вакансий ещё не завершён из-за архивации, `responsesCount` и пагинации;
  - главная страница проекта не использует вакансии как завершённый пользовательский сценарий.
- Признаки техдолга / нестабильности:
  - транзакция добавлена в составной update-flow вакансии, но в других составных операциях транзакционность остаётся неполной;
  - часть полей и derived-data не оформлена как устойчивый response contract;
  - нет пагинации для списков вакансий.

### 3.6 Отклики

- Статус: `частично реализовано`
- Оценка:
  - Кандидат может создать отклик и увидеть свой отклик по вакансии.
  - Рекрутер может получить список откликов по вакансии.
  - В отклике есть сообщения и визуализация статуса.
- Что подтверждается кодом:
  - `POST /api/vacancies/:id/applications`;
  - `GET /api/vacancies/:id/applications`;
  - `GET /api/vacancies/:id/applications/me`;
  - экран отклика кандидата;
  - экран списка откликов рекрутера;
  - базовое отображение сообщений отклика.
- Почему не выглядит завершённым:
  - frontend и backend не состыкованы по полю текста отклика;
  - нет backend-операций для отклонения, принятия, продвижения по этапу, отправки новых сообщений;
  - модалка отказа существует только как интерфейс без действия;
  - recruiter-side action buttons являются недоделанным интерфейсом.
- Признаки техдолга / нестабильности:
  - составная операция создания отклика без транзакции;
  - контракт ответа и сообщений не отделён от сущностей;
  - поведение recruiter workflow фактически не завершено.

### 3.7 Воронка найма

- Статус: `частично реализовано`
- Оценка:
  - Есть доменная модель шага воронки и форма её редактирования в вакансии.
  - Есть UI-визуализация прогресса по шагам в отклике.
- Что подтверждается кодом:
  - хранение `funnelSteps`;
  - upsert шагов при создании и обновлении вакансии;
  - поля `approveMessage`, `rejectMessage`, `shouldCreateCall`;
  - отображение текущего статуса отклика относительно шагов.
- Почему не выглядит завершённым:
  - нет реального backend-flow для перевода отклика между шагами;
  - шаблоны сообщений сохраняются, но не применяются;
  - флаг создания звонка сохраняется, но не используется.
- Признаки техдолга / нестабильности:
  - поведение воронки частично есть в UI и модели, но не доведено до исполняемого процесса;
  - нет подтверждённого business flow для final approval/rejection.

### 3.8 Вложения и медиа

- Статус: `частично реализовано`
- Оценка:
  - Attachment-модель используется для аватара кандидата и логотипа компании.
  - Backend умеет хранить и отдавать бинарный контент.
- Что подтверждается кодом:
  - создание attachment через внутренние сервисы;

- Почему не выглядит завершённым:
  - URL вложения жёстко зашит на `http://localhost:8000`;
  - нет видимой централизованной env-настройки для attachment base URL;
  - не видно завершённой стратегии валидации mime type.
- Признаки техдолга / нестабильности:
  - хардкод окружения;
  - частично неформализованная валидация входящих файлов.

### 3.9 Главная страница, shell и навигация

## 4. Front / Back несостыковки

### 4.1 Существенные расхождения

- `responsesCount`:
  - frontend ожидает количество откликов в карточке вакансии;
  - backend явно не описывает и не заполняет это поле.

- Навыки кандидата:
  - backend поддерживает `skillIds` у кандидата;
  - frontend-форма кандидата этого не даёт.

- Навыки вакансии:
  - backend поддерживает `skillIds` у вакансии;
  - во frontend-форме вакансии не видно пользовательского выбора навыков.

### 4.2 Расхождения уровня поведения

- Города и специализации:
  - backend предоставляет реальные справочники;
  - frontend при ошибке silently подставляет моки.

- Статусы и workflow отклика:
  - frontend показывает действия recruiter workflow;
  - backend не предоставляет соответствующих endpoint и логики.

## 5. Что выглядит недоделанным

- Главная страница приложения.
- Смена пароля рекрутера.
- Архивация и разархивация вакансии.
- Recruiter-side действия по отклику:
  - отказ;
  - принятие;
  - переход на следующий этап;
  - отправка сообщений.
- Исполнение воронки найма, а не только её хранение и отображение.
- Использование навыков в реальном frontend-сценарии кандидата и вакансии.
- Полная production-конфигурация auth/cors/attachment URLs.

## 6. Технический долг и риски

### 6.1 Backend

- `TypeORM synchronize: true` включён, миграции не настроены.
- Много составных операций без транзакций:
  - создание / обновление кандидата;
  - создание / обновление рекрутера;
  - создание / обновление вакансии;
  - создание отклика.
- Часть ошибок оформлена как временные или неунифицированные исключения.
- Нет стабильного response layer, ответы часто равны текущей форме entity и joins.

### 6.2 Frontend

- Хардкод backend URL для attachments.
- Моки для городов и специализаций скрывают фактические ошибки API.
- Есть UI-кнопки и модалки, не доведённые до действия.
- Часть полей формы валидируется упрощённо.

### 6.3 Интеграционные риски
- Риск некорректного обновления связанных полей вакансии снижен после стабилизации update-flow, но не устранён полностью для остальных составных операций.
- Часть frontend-функционала создает впечатление завершённости, но по коду не доведена до backend execution path.

## 7. Неопределённости

- Нельзя надёжно определить, какие из текущих недоделанных UI-элементов считались запланированными ближайшими фичами, а какие являются ранними заготовками.
- По коду нельзя надёжно определить целевой бизнес-процесс recruiter workflow после получения отклика:
  - кто и как двигает по шагам;
  - когда выставляется `approved` или `rejected`;
  - как должны использоваться шаблоны сообщений и `shouldCreateCall`.
- Не удалось подтвердить текущее состояние сборки и тестов в локальном окружении:
  - команда `yarn` отсутствует;
  - `corepack yarn` не запустился из-за ошибки прав на cache.

## 8. Приоритетные темы для следующего этапа планирования

1. Зафиксировать и исправить реальные frontend/backend расхождения по контрактам.
2. Определить целевой workflow отклика и воронки найма как сквозной пользовательский сценарий.
3. Довести вакансии до целостного состояния:
   - обновление связей;
   - архивация;
   - количество откликов;
   - при необходимости пагинация.
4. Решить, должны ли навыки быть частью MVP-потока кандидата и вакансии.
5. Убрать инфраструктурные риски:
   - migrations вместо `synchronize`;
   - env-driven config;
   - транзакции в составных операциях.
